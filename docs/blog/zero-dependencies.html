A Rust UI framework with zero dependencies

Dija is a cross-platform UI framework I'm building in Rust. It runs on macOS, Linux, iOS, Android, and terminals — same codebase, same binary. It has zero external dependencies.

Not "few dependencies." Not "minimal dependencies." Zero.

The entire runtime — the layout engine, the GPU pipeline, the font parser, the actor system, the coroutine scheduler — is built from scratch. cargo build downloads nothing. Here's why, and what that actually cost.

The constraint

I'm also building an operating system called zelez OS. On zelez, every app is a WASM module running on bare metal. There is no package manager. There is no std. If a crate depends on anything I can't compile to #![no_std], it can't ship.

This wasn't a philosophical decision about software purity. It was a survival constraint. I needed a GUI framework that could run on a 256KB microcontroller and a desktop workstation, from the same codebase. Nothing like that existed, so I started building one.

Early on I tried pulling in a few crates. Each one brought transitive dependencies. Some needed std. Some assumed an allocator I didn't have. Some were fine on desktop but broke on embedded. After fighting this for a few weeks, I made the decision: zero deps, and I'll build what I need.

That was about a year ago. Here's what "building what I need" actually looked like.

A TTF font parser

Most Rust projects use fontdb + rusttype or call into FreeType and HarfBuzz through FFI. None of those work in no_std. So I wrote a TrueType parser from scratch.

It parses the tables you need for basic text rendering: head, hhea, maxp, cmap (format 4 for BMP, format 12 for full Unicode), loca, glyf, hmtx, and kern. It handles compound glyphs with recursive component expansion. It does pair-based kerning.

pub struct Font {
    data: Vec<u8>,
    units_per_em: u16,
    ascender: i16,
    descender: i16,
    line_gap: i16,
    num_glyphs: u16,
    // Table offsets
    glyf_offset: usize,
    loca_offset: usize,
    hmtx_offset: usize,
    loca_format: i16, // 0 = short, 1 = long
    // Kern table
    kern_pairs_offset: usize,
    kern_num_pairs: u16,
    // Cmap subtables
    cmap4: Option<Cmap4>,   // BMP
    cmap12: Option<Cmap12>, // Full Unicode
    // Glyph cache (LRU eviction for MCU)
    cache: GlyphCache,
}

The glyph rasterizer flattens quadratic Bézier curves into line segments, then does scanline coverage accumulation. Coverage is stored as linear (not sRGB-encoded) so that GPU bilinear filtering and color * coverage compositing produce correct blending at glyph edges.

On desktop, the glyph cache is unbounded. On MCU, you can set a limit:

// Desktop: unlimited cache
let cache = GlyphCache::new();

// MCU with 32KB for glyphs
let cache = GlyphCache::with_limit(256);

When the cache is full, the least-recently-used entry is evicted. Same code, same API, different memory budget.

What's missing vs HarfBuzz: complex text shaping. Arabic, Devanagari, and other scripts that need contextual glyph substitution don't work yet. For Latin, CJK, and most other scripts, it's fine. This is a known gap, and I'll address it — but reimplementing OpenType Layout tables is a project on its own.

Was it worth it? The parser is about 1,000 lines. It took maybe two weeks. For what Dija needs today, it's enough. And it runs on a microcontroller, which FreeType never will.

Raw Metal and Vulkan FFI

Most Rust GPU projects use wgpu (a cross-platform abstraction) or ash (Vulkan bindings) or metal-rs (Metal bindings). I use none of them. Dija talks to Metal and Vulkan directly through raw FFI.

For Metal, that means writing Objective-C FFI by hand:

pub type Id = *mut c_void;
pub type Sel = *mut c_void;

#[link(name = "objc", kind = "dylib")]
unsafe extern "C" {
    pub fn objc_getClass(name: *const u8) -> Class;
    pub fn sel_registerName(name: *const u8) -> Sel;
    pub fn objc_msgSend();
}

#[link(name = "Metal", kind = "framework")]
unsafe extern "C" {
    pub fn MTLCreateSystemDefaultDevice() -> Id;
}

/// Create an NSString from a null-terminated byte slice.
pub fn nsstring(s: &[u8]) -> Id {
    let f: unsafe extern "C" fn(Id, Sel, *const u8) -> Id =
        unsafe { core::mem::transmute(objc_msgSend as *const c_void) };
    unsafe {
        f(
            cls(b"NSString\0") as Id,
            sel(b"stringWithUTF8String:\0"),
            s.as_ptr(),
        )
    }
}

Every Metal API call goes through objc_msgSend with transmuted function pointers. It's ugly. It's also about 640KB of Rust code, covering device creation, command queues, render and compute pipelines, texture management, MSL shader compilation, and MSAA.

For Vulkan, it's a similar story: raw FFI bindings with 34 pre-compiled SPIR-V shader modules.

Why not wgpu? Three reasons. First, wgpu brings a large dependency tree — it's an abstraction layer over Metal, Vulkan, DX12, and WebGPU, and most of that abstraction is weight I don't need. Second, wgpu targets a common denominator across backends. I want to use Metal-specific features on macOS and Vulkan-specific features on Linux without fighting an abstraction. Third, wgpu doesn't run in no_std.

The tradeoff: I maintain two GPU backends instead of one abstraction. When Metal changes an API or Vulkan adds a feature, I update my bindings directly. There's no community of thousands finding bugs for me. I've accepted that. For a framework that needs to run on bare metal, owning the GPU layer isn't optional — it's the whole point.

A flexbox and CSS grid layout engine

Layout is the heart of a UI framework. Most Rust projects use taffy (the successor to stretch) or bind to Facebook's yoga. Both are good. Neither is no_std compatible without modification, and both bring their own allocation strategies that don't align with what I needed.

Dija's layout engine implements CSS Flexbox Level 1 (section 9) and CSS Grid with fractional units, auto-placement, column/row spanning, and masonry layout. It's arena-allocated — every layout computation uses a bump allocator that gets reset each frame, producing zero heap fragmentation.

pub(crate) struct FlexItem {
    pub child_index: usize,
    pub base_size: Scalar,
    pub min_main: Scalar,
    pub max_main: Scalar,
    pub flex_grow: Scalar,
    pub flex_shrink: Scalar,
    pub final_main: Scalar,
    pub final_cross: Scalar,
    pub main_pos: Scalar,
    pub cross_pos: Scalar,
    pub frozen: bool,
}

pub(crate) struct FlexLine {
    pub start: usize,
    pub end: usize,
    pub main_size: Scalar,
    pub cross_size: Scalar,
    pub cross_pos: Scalar,
}

The layout cache fingerprints each subtree. If a container's constraints haven't changed since last frame, the entire subtree is skipped. On a dashboard with 200+ nodes, this cuts layout time by 60-80%.

Was it worth it? The layout engine is probably the piece I'm most glad I built myself. It's tightly integrated with the rendering pipeline, it uses the arena allocator I designed for the framework, and I can add features (like masonry) without waiting for upstream.

Coroutines in assembly

Dija's runtime is built on an Erlang-inspired actor model. Every view is an actor with its own mailbox and lifecycle. The scheduler needs to suspend and resume thousands of actors cheaply — much cheaper than OS threads, and without the overhead of Rust's async machinery.

So I wrote coroutines. In assembly. For five architectures.

Here's x86_64:

#[repr(C)]
pub struct CpuContext {
    rsp: u64,
    rbx: u64,
    rbp: u64,
    r12: u64,
    r13: u64,
    r14: u64,
    r15: u64,
}

#[unsafe(naked)]
pub unsafe extern "C" fn context_switch(
    from: &mut CpuContext,
    to: &CpuContext,
) {
    core::arch::naked_asm!(
        // Save callee-saved registers
        "mov [rdi + 0x00], rsp",
        "mov [rdi + 0x08], rbx",
        "mov [rdi + 0x10], rbp",
        "mov [rdi + 0x18], r12",
        "mov [rdi + 0x20], r13",
        "mov [rdi + 0x28], r14",
        "mov [rdi + 0x30], r15",
        // Restore callee-saved registers
        "mov rsp, [rsi + 0x00]",
        "mov rbx, [rsi + 0x08]",
        "mov rbp, [rsi + 0x10]",
        "mov r12, [rsi + 0x18]",
        "mov r13, [rsi + 0x20]",
        "mov r14, [rsi + 0x28]",
        "mov r15, [rsi + 0x30]",
        "ret",
    );
}

Save seven registers. Restore seven registers. ret. That's a context switch. The same pattern is implemented for ARM64, RISC-V (64 and 32-bit), and ARM Thumb.

A trampoline sets up the initial context so that the first context_switch into a new coroutine jumps to its entry function:

pub unsafe fn init_context(
    ctx: &mut CpuContext,
    stack_top: *mut u8,
    entry: unsafe fn(*mut u8),
    data: *mut u8,
) {
    let mut sp = (stack_top as u64) & !0xF; // 16-byte alignment
    sp -= 8;
    unsafe { *(sp as *mut u64) = 0 };       // null return address
    sp -= 8;
    unsafe { *(sp as *mut u64) = trampoline as u64 };
    ctx.rsp = sp;
    ctx.r12 = entry as u64;   // entry point
    ctx.r13 = data as u64;    // data pointer
}

The entry and data pointers are stashed in callee-saved registers (r12, r13) so they survive the context switch. The trampoline reads them and calls entry(data).

Why not tokio or async/await? Tokio is designed for async I/O — network servers, file operations, timers. Dija needs cooperative multitasking for actors, not I/O multiplexing. The actor system needs to suspend an actor mid-computation, not mid-await. And tokio doesn't work in no_std.

The tradeoff: I'm maintaining assembly code for five architectures. When a new platform appears, I write a new context switch stub. It's about 50 lines per architecture. I've tested it with Miri (where possible) and with extensive property-based tests. It's one of the more nerve-wracking pieces to maintain, but it's also one of the smallest.

What I didn't build

I want to be honest about where the line is.

libm (sin, cos, sqrt): Dija depends on the libm crate — a pure-Rust, no_std-compatible implementation of the C math library. I considered writing my own, but libm is well-tested, platform-independent, and tiny. Reimplementing it would have been vanity, not engineering.

syn, quote, proc-macro2: Dija's proc macros (the ui! and comp! macros) depend on the standard proc-macro toolchain. These are build-time dependencies — they run at compile time and produce code that ships with zero runtime deps. The binary you run has no trace of them.

Zero runtime dependencies doesn't mean zero build dependencies. I draw the line at what ships in the binary.

Was it worth it?

Honestly? It depends on what you're optimizing for.

What's worse: I have less community testing than battle-tested crates. My TTF parser has been tested by one person (me) on a few hundred fonts. HarfBuzz has been tested by billions of web pages. My Vulkan bindings haven't been beaten on by thousands of game developers. I find bugs that ash found years ago.

Feature velocity is slower. Every missing feature is something I have to build, not import. When someone asks "does Dija support X?" the answer is sometimes "not yet, because I had to write Y first."

What's better: One binary. Five platforms. Total control over the stack. No supply chain risk — I can audit every line. Compile times are predictable because there's nothing to download. Binary size is small because there's no dead code from crates I half-use.

And it runs on a 256KB microcontroller. That was the whole point.

The real cost: Time. I spent months building infrastructure that could have been features. The TTF parser, the GPU backends, the layout engine, the coroutines — that's probably six months of work that users never see directly. They see a dashboard rendering on five platforms. They don't see the 142,000 lines underneath.

Would I do it again? Yes. But I wouldn't recommend it to most people. This approach only makes sense if you have a hard constraint (like bare metal) that forces it, or if you're building a long-lived foundation where owning the stack pays compounding returns.

Dija is both.
